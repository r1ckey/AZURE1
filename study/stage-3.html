Microsoft Entra ID (旧 Azure AD)

  • ID 実装: クラウドベースの ID およびアクセス管理サービス。
  • テナント: 組織ごとの専用インスタンス。
  • 条件付きアクセス: 「特定の場所からのみ接続可能」といった高度なセキュリティ制御。

Azure のセキュリティと ID 管理の心臓部が Microsoft Entra ID です。ユーザーの認証だけでなく、ディレクトリ内のグループ管理、外部ゲストユーザー（B2B）の招待、そして多要素認証 (MFA) など、組織全体の安全を守る要となります。

RBAC (ロールベースのアクセス制御)

  • 所有者 (Owner): 全ての権限。アクセス権の付与も可能。
  • 共同作成者 (Contributor): リソースの作成・変更はできるが、他者への権限付与はできない。
  • 閲覧者 (Reader): 閲覧のみ。
  • カスタムロール: 必要な権限だけを組み合わせた独自の役割を作成可能。

「誰がどこまで操作できるか」を定義するのが RBAC です。Azure では、管理グループ、サブスクリプション、リソースグループといった階層構造に基づいて権限が継承されます。

マネージド ID (Managed Identities)

  • パスワード不要: リソース（VM 等）自体に ID を持たせ、他のサービスへ安全にアクセス。
  • システム割り当て: 特定のリソースに紐付く、1対1の ID。
  • ユーザー割り当て: 独立したリソースとして作成し、複数の VM 等で共有可能な ID。

アプリケーションから Azure Key Vault や SQL DB に接続する際、プログラム内に接続文字列や ID/PW を書き込む必要がなくなる、画期的な機能です。Azure 基盤が ID を自動管理するため、パスワード漏洩のリスクを極限まで減らします。

Azure Policy とリソースロック

  • Azure Policy: リソースの設定ルールを強制（例: 公開 S3 の禁止）。
  • リソースロック: 「削除不可」や「読み取り専用」の状態にして、誤操作を防止。

Azure Policy はリソースがデプロイされた後も動的に「ルール違反」がないか監視し、必要であれば作成自体を拒否します。一方、リソースロック (Locks) は、管理者であっても重要な本番サーバーを「うっかり削除」できないように物理的な『鍵』をかけるイメージです。
