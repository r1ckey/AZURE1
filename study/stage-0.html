Azure の管理階層構造

  • 管理グループ: 複数のサブスクリプションを束ねる最上位。
  • サブスクリプション: 課金とアクセスの単位。
  • リソースグループ (RG): 関連するリソースを論理的にまとめる箱。

Azure の最大の特徴は、この強固な階層構造にあります。リソース ＞ リソースグループ ＞ サブスクリプション ＞ 管理グループという順に大きくなり、上位で設定したポリシーやアクセス権（RBAC）は下位へ自動的に継承されます。

グローバルインフラストラクチャ

  • リージョン: 地理的な境界。東日本、西日本など。
  • 可用性ゾーン (AZ): リージョン内の独立したデータセンター群。
  • リージョンペア: 300マイル以上離れたリージョン同士のペア（災害対策）。

Azure は世界中に 60 以上のリージョンを展開しています。各リージョン内には、電源や冷却が独立した複数の可用性ゾーン (AZ) が存在し、これらを活用することで高い可用性を実現します。

リージョンペア（例: 東日本と西日本）を活用すれば、万が一大規模な災害が発生しても、自動的に他方のリージョンへデータを守る体制が構築できます。

Azure Resource Manager (ARM)

  • 単一の入り口: 全ての操作（ポータル、CLI、SDK）は ARM 経由で行われる。
  • 宣言型管理: ARM テンプレート (JSON/Bicep) により、コードでインフラを定義。

ARM は Azure の管理プレーンの頭脳です。あなたがマウスでリソースを作成しても、コマンドラインで実行しても、全て ARM が受け取って一貫性のある方法で処理します。

ガバナンスとアクセス管理の基本

  • RBAC: 「誰が」「何を」できるかを役割（ロール）で管理。
  • Azure Policy: リソースの「あるべき姿」を強制（例: インスタンスサイズの禁止）。
  • Azure Blueprints: 環境全体（RG、Policy、RBAC、ARM）のひな型。

Azure でのアクセス管理は RBAC (Role-Based Access Control) を中心に行います。また、設定ミスによるセキュリティ事故を防ぐために、あらかじめ「東京リージョン以外には作成禁止」といったルールを Azure Policy で定義しておくことがエンタープライズ利用では必須となります。
